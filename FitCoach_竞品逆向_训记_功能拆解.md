> 说明：以下结论来自对 1m56s 录屏的关键帧抽取与时间线还原；重点不是“它有哪些页面”，而是它用什么产品逻辑与交互组织，去满足训练场景下的**心理连续性**与**高频记录效率**。

# Part 1 — 该 App 的核心产品定位

## 它真正解决什么问题？
不是“给训练计划”，而是解决**健身训练的可量化、可复盘、可持续优化**：
- 把纸笔/Excel 的训练记录数字化（避免丢失、避免零散、可长期积累与回顾）。
- 用“训练容量=组数×重量×次数”等指标把训练从“感觉”推进到“证据”（数据驱动自我纠偏）。

## 它主要吸引哪类用户？
典型是“已经在健身、并愿意为训练质量付出记录成本”的用户（偏硬核）：
- 明确知道自己在练什么、为什么练、练得如何。
- 想要自由自定义（计划、动作、输入）而不是被课程/教练牵着走。
- 对图表、历史复盘、长期进步有强动机。

## 它核心卖点是什么？
1. **足够自由（不指手画脚）**：用户想怎么练就怎么练，想怎么记就怎么记（训练主权在用户）。  
2. **训练会话的“组级别记录”**：每完成一组打勾一次，把“训练中最小事实”落盘，形成节奏与确定性。  
3. **把“开练门槛”拆成两套入口**：  
   - “创建一个空的训练” → 立即计时，给不做计划的人直接开练。  
   - “计划模板” → 给多数“提前规定好训练内容”的用户，避免每次重建动作/组数/次数。  
4. **数据资产化**：历史/图表/容量等，把每次训练变成可复用的长期资产。

## 它为什么能让用户留下来？
它把留存的核心从“内容供给”换成“**训练资产沉没成本 + 自我提升反馈**”：
- 记录越久，历史越完整 → 训练决策越来越依赖它 → 越不容易迁移/卸载。
- 每组打勾→容量累积→图表可见→形成“我在进步”的可视证据。
- 日历编排/颜色标记/模板复用 → 把训练变成“可执行的日程系统”，降低每日启动成本。

# Part 2 — 用户行为模型

## 用户打开 App 后第一目标是什么？
  1) **现场开练型**：立刻开始一次训练并计时（空训练入口）。  
  2) **计划执行型**：今天练什么？直接从模板/日历里点“开始训练”。

## 用户最频繁行为是什么？
- 训练中：**输入重量/次数 → 打勾完成一组 → 休息计时 → 下一组**。
- 训练外：复盘历史/图表、调整模板（把未来训练预先结构化）。

## 用户停留时间最长在哪？
- “训练会话页”（高频交互密度最高）。
- “历史/图表”（训练后的自我奖励与策略调整的主要舞台）。

## 用户最容易形成习惯的点是什么？
- **最小行为闭环**：做完一组 → 打勾 → 容量累积（即时确定性 + 微奖励）。  
- **日历/模板把训练前的犹豫前置消化**：今天点一下就开练（把决策从健身房现场移走）。

## 用户为什么愿意每天打开？
- “我要执行今天的训练”与“我要看到进步证据”这两个动机，分别由：
  - 日历/模板（任务触发）
  - 容量/图表（奖励反馈）
共同驱动。

## 用户为什么不容易卸载？
- 训练记录具有不可替代的个人意义（长期资产），且迁移成本极高。
- 外部同步（健康平台/手表）进一步绑定生态（虽不是核心，但增强沉没成本）。

# Part 3 — 训练记录流分析（重点）

按“进入训练 → 选择动作 → 输入组数 → 休息 → 下一动作 → 完成训练”拆解：

## 进入训练
- **空训练**：点击即开始计时，属于“把训练启动当作 runtime 事件”的设计（非常贴近 FitCoach 的 instant-entry 理念）。
- **模板训练**：解决“绝大部分人训练前已规定内容”的现实，避免在健身房现场做结构搭建。

## 选择动作
- 录屏显示：按部位 + 器械标签筛选，列表卡片化；可置顶/历史复用（降低搜索成本）。

## 输入组数（核心）
- 以“组”为主语：每组一行，输入重量/次数，右侧 ✓ 完成。
- “打勾”不仅是 UI 反馈，还会驱动容量计算（把交互动作绑定到训练指标更新）。

## 休息
- 录屏出现组间计时/组间有氧弹窗，可最小化、可震动——把“休息”从无聊等待变成受控工具，且不阻塞训练记录流。

## 下一动作 / 完成训练
- 训练页内基本不需要跳转（优势），但跨模块存在加载遮罩导致断流风险（劣势）。

### 关键结论：它是否真正“减少了训练中的思考”？
- 对“硬核用户”是减少的：因为他们本来就有计划/动作库/训练概念，训记做的是把这些概念映射到一个稳定输入路径。  
- 但对“轻量用户”反而可能增加思考：自由与工具箱带来更多决策点（热身组/递减/RPE/模板体系）。

# Part 4 — 信息架构分析

## 首页核心信息是什么？
- “今天练什么 / 正在练什么 / 进度如何”优先级高：
  - 今日训练卡片（含进度百分比、GO、继续训练锚点）
  - 计划/模板/周期（把训练当作日程系统）

## 它最强调什么？（产品哲学）
- **训练是一个可规划、可执行、可复盘的工程系统**（偏计划+数据导向）。
- 不是 AI 导向、也不是社交导向；它的“专业感”来自记录粒度与统计能力。

## 是否偏“计划导向 / 数据导向 / 仪表盘导向”？
- 明显偏：**计划导向 + 数据导向**  
  - 计划：模板、日历、配色标记、快捷方式
  - 数据：容量、历史、图表
  - 仪表盘：作为结果呈现存在，但不是核心叙事

# Part 5 — UX 与视觉策略分析

## 为什么用户会觉得专业？
- 专业感不是“花哨 UI”，而是：
  - 支持组级记录（符合真实训练记录习惯）
  - 强指标（容量/历史/图表）让用户觉得“这是给认真训练的人用的”
  - 动作库/3D/讲解强化“可信度”（即使训练中未必常看）

## 为什么用户会觉得容易上手？
- “空训练”入口把新手最害怕的“先建计划/先选课程”拿掉了：先开练再说。
- 训练页的核心动作很少：填数字 + 打勾 + 计时。

## 是否使用大量卡片 / 是否存在视觉疲劳？
- 卡片化明显、信息密度偏高；长时间浏览“计划列表/数据页”有疲劳风险。
- 但对它的核心人群（硬核、愿意看数据）来说，密度本身是价值的一部分。

## 训练时是否容易误触？是否适合 gym 快速查看？
- 大块点击区（组行、✓）适合单手与出汗场景；  
- 风险来自：训练页功能按钮较多（设置/更多/高级输入），新手可能误入“工具箱”。

# Part 6 — 工程与前端架构逆向分析

## 技术栈推测
- 仅从安卓包名线索（第三方应用页显示 `com.trainnote.rn`）推断：**React Native 可能性极高**。

## 是否依赖重客户端状态？是否 local-first？
- 必然重客户端状态：训练计时、逐组输入、最小化恢复、组间计时都必须本地可用。
- 训练过程很可能是 local-first（SQLite/Realm/AsyncStorage 等）+ 后台同步（健康平台/云端）。

## optimistic update / append-only 的可能性
- 训练输入必须即时回显，典型会采用 optimistic update。
- 组勾选与容量更新非常适合 event log（append-only）建模：SetEdited/SetCompleted/RestStarted/RestEnded…

## hydration blocking / 大组件 / 重渲染风险
- 录屏中出现多次遮罩 Loading，说明：
  - 部分页面仍是“请求完成→整页渲染”的阻塞式数据流；
  - 如果 RN 页面把多个模块绑在同一个 store，输入可能触发不必要的重渲（高刷机更敏感）。

# Part 8 — 竞品真正的优势（非常重要）

## 它真正比 FitCoach 强的地方（当前可见）
1. **“计划模板 + 日历编排 + 颜色标记”把训练变成可执行日程**  
   - FitCoach 去计划化是方向优势，但也会损失“长期编排与复盘的可视结构”。  
2. **容量指标叙事更强**（打勾→容量→图表）  
   - 它把每个微操作都变成指标增量，形成强反馈闭环。

## 为什么用户会偏爱它？
- 它让硬核用户获得两种确定性：
  - **训练过程确定性**：每组打勾就是事实落盘
  - **长期进步确定性**：图表/容量把努力变成证据

## 它形成产品壁垒的地方
- 壁垒不是功能，而是“训练资产的可复盘性”与“记录习惯”：
  - 模板/日历降低每日启动成本
  - 历史/图表强化长期价值

## 哪些是“感知优势” vs “真实效率优势”？
- 感知优势：动作库内容、3D/视频讲解、专业术语与密集选项（显得硬核）。  
- 真实效率优势：空训练直达、组级 ✓、休息工具、模板复用减少重建成本。  
- 视觉包装：卡片化与“专业数据页”的仪表盘外观（有用，但不是决定性的效率来源）。

# Part 9 — FitCoach 可以吸收什么

## 可以直接学习
1. **空训练（Instant Workout）作为一级入口**：点击即创建会话并计时（与你们 instant-entry 完全同向）。  
2. **组级 ✓ 的强 commit 语义**：把“我完成了”具象化为一个确定事件，减少记错/漏记焦虑。  
3. **训练进行中回流锚点（全局 presence）**：允许用户切后台/切页面但不丢 session。

## 需要改造后学习
1. **模板/日历/配色**：  
   - 学习的是“减少每日启动成本”，不是把 FitCoach 变成计划 App。  
   - 建议做成“预测与自动推进的背景结构”（轻计划），而不是让用户维护模板。
2. **容量指标叙事**：  
   - FitCoach 可把容量/强度/频次等作为“训练流的实时副产物”，用 streaming 方式渐进呈现；不要把用户带去重仪表盘页。

## 不应该学习
1. **“自由=把所有复杂性都暴露给用户”**  
   - FitCoach 的优势恰恰是“训练中少思考”，应把复杂性留给系统（预测/自动化），而不是增加工具箱选项。
2. **全家桶信息架构**（饮食/经期/体测作为主导航中心）  
   - 会稀释 FitCoach 的训练 runtime 心智，破坏 streaming 首页的主线。

# Part 10 — FitCoach 下一步升级方向（重点，可落地）

## UX 升级
1. **一级入口：Start Empty Workout（默认） + Continue Workout（若存在）**
   - 打开 App 的首要 CTA 永远是“开始/继续训练”，其他内容 streaming 下沉。
2. **把“✓”做成训练流的核心节拍器**
   - 每组完成：✓（带 haptic）→ 自动触发 rest overlay → rest 结束给下一组预测。
3. **模板替代方案：Auto-Template（隐形模板）**
   - 用户不需要创建模板；系统根据最近 N 次训练自动生成“默认序列”，用户只需开始练。

## Runtime 升级（runtime-first）
1. **Append-only event log 做训练事实源**
   - UI 写事件 → runtime reducer 产出 snapshot → UI islands 订阅局部 slice。  
2. **全局 Session Presence**
   - 训练进行中 pill 由 runtime 直接驱动，独立于路由与页面生命周期（避免“页面卸载=训练丢失”）。

## 训练流升级
1. **预测驱动的“下一步”**
   - 训记通过模板减少重建；FitCoach 用预测把下一动作/下一组直接放到用户拇指下。
2. **休息作为非阻塞工具**
   - rest overlay 必须是 non-blocking：可最小化、可跨页面存在、可离线运行。

## 状态流升级
1. **显式状态机 + 可恢复**
   - idle → active → minimized → resumed → completed → sync_pending/synced  
2. **每次 ✓ 都是持久化点**
   - 把“完成训练”从单点 commit 降级为“会话关闭”，降低用户忘点完成的风险。

## 首页升级（streaming UI）
1. **首页首屏只渲染 3 个岛**
   - Continue/Start、Next Step（预测）、Recent History（可复用动作）
2. **其余模块渐进加载**
   - 推荐、统计、计划视图都必须是“可中断的流”，不能阻塞训练入口。

## 高频输入优化
1. **微调优先于输入**
   - +2.5kg / +1rep / RPE slider 这种“微调控件”应比完整键盘更先出现。
2. **一键继承上一组**
   - 训练中最常见行为：沿用上一组重量次数并 ✓；把它做成默认操作。

## 动作复用优化
1. **最近动作=默认动作库**
   - 动作库搜索是长尾；训练中应优先呈现“最近/同周期”候选，降低检索成本。

# Part 11 — 最终产品结论

1) **为什么这个竞品能成功？**  
它用“空训练直达 + 组级打勾 + 容量/图表复盘 + 模板/日历降低启动成本”把训练变成一个可执行、可量化、可持续优化的系统。

2) **它真正抓住了什么用户心理？**  
硬核用户的核心不是“被教练带”，而是“我在掌控并在进步”：**自由主权 + 数据证据**带来确定性与长期满足。

3) **它最大的产品优势是什么？**  
把训练的最小事实单位（组）与关键指标（容量）绑定到一个极稳定的交互（✓），形成高频闭环与长期复盘资产。

4) **它最大的架构问题可能是什么？**  
多模块/重页面导致遮罩 Loading 与断流；若 RN 大 store 驱动整页，输入与高刷下更易出现重渲染与掉帧。

5) **为什么 FitCoach 有机会做得更强？**  
FitCoach 可以用 runtime-first + instant-shell + streaming/island 把“训记的闭环优势”升级为“系统级不断流”：  
用 predictive set 把输入压缩成确认/微调，用 append-only 事件流保证可恢复与高刷，把计划与复盘做成训练流的渐进副产物，而不是训练的前置门槛。

# Part 1 — 功能表层描述


**这个功能是什么**
- 这是一个“训练日志 + 动作库 + 饮食/体测/经期 + 可穿戴/健康平台同步”的综合型健身记录 App（录屏显示品牌为「训记」）。

**用户在做什么**
- 用户创建/继续一次训练 → 从动作库选动作 → 逐组填写重量/次数并打勾完成 → 使用组间计时与“组间有氧” → 可随时最小化离开 App，保持训练进行中。

**核心目标是什么**
- 在健身房高频场景下，把“训练过程中的动作/组数数据”以尽可能低摩擦的方式落盘，同时用“计划/模板/进度/数据图表/外部同步”增强留存。

# Part 2 — 用户流程拆解（时间线）

## Entry Trigger（用户从哪里进入？）
录屏出现了多个入口（体现其产品策略：**入口冗余=降低开练门槛**）：
1. 首页（训练/饮食聚合页）里的「今日训练」卡片：有 “GO!” / 进度百分比。
2. 底部栏的快捷创建（中心“X/浮层”）：弹出一排快捷项（新建训练、新建有氧、新建 HIIT、新建休息、身体数据、经期记录）。
3. 若训练已开始：底部/页面右下出现「训练进行中 00:00」的悬浮提示，作为“继续训练”的回流锚点。

## First Interaction（第一步操作是什么？）
典型路径（录屏覆盖到）：
1. 进入训练会话页（顶部计时 00:00，右上「完成」）
2. 点击底部「+ 加动作」
3. 进入动作库：按部位（胸/背/腿…）+ 器械标签（杠铃/哑铃…）筛选，卡片式动作列表，可“置顶”

## Main Loop（用户主要循环行为是什么？）
循环的最小单元是“**一组**”而不是“一个动作”：
1. 在训练页看到动作卡片 → 下方是组表格（第1组、第2组…）
2. 点重量/次数格子 → 弹出自定义数字键盘（含 kg/lbs、热身组/递减组、RPE、一键改+/-、加一组等）
3. 输入完成 → 点每组右侧勾选（✓）标记完成
4. 触发组间休息/组间有氧：弹出环形计时弹窗，可手机震动、可最小化
5. 重复：下一组 / 下一个动作

## Completion State（用户如何结束？）
录屏未展示完成后的总结页，但从信息结构可推断：
- 顶部「完成」是会话结束的明确 commit 操作（很可能进入：训练总结/可分享/写心得/保存模板等）。
- 即使没完成，也能最小化离开，后续通过「训练进行中」回到会话（**把“完成”从强制流程变成可选**，降低中断成本）。

## Retention Hook（它如何促使用户继续使用？）
录屏中明确露出的留存钩子：
- 「今日训练」进度百分比（例如 20%）：把训练拆成可完成的进度条任务。
- 「今日安排」里出现“周期结束/模板/无训练”等列表：用“计划与周期”制造连续性。
- 动作详情里的「历史 / 图表」：把每次训练变成可回看的长期数据资产。
- 外部同步（华为运动健康、手表支持、微信生态等）：把数据嵌入更大系统，增加迁移成本。
- Pro 订阅标签遍布关键卡片：功能分层+订阅转化。

# Part 3 — UX 设计分析（训练中是否被打断？）

## 1) 是否存在等待感 / 页面阻塞？
有，且比较“传统”：
- 多处出现全屏/遮罩 Loading（例如“加载中”、黑底进度 38%）——典型的**请求-渲染串行**。
- 这类等待对“训练中即时记录”的心理连续性伤害大：用户在喘/急/单手时不愿意等。

## 2) 是否有页面阻塞/多余点击？
训练页内相对少跳转（这是它做对的点），但功能层面仍有“设置/模板/统计/身体数据”等跨模块跳转：
- 对“开练→记录→继续”的主链路来说，这些模块是**长尾**，容易造成“探索型点击”，训练中断流。

## 3) 是否存在认知负担？
偏高（但给高阶用户很爽）：
- 自定义键盘把高级概念（热身组、递减组、RPE、一键改）集中塞进同一层，信息密度大。
- 对新手而言，可能形成“我要不要用这些功能”的额外决策；对老手则是效率工具箱。

## 4) 是否符合训练中单手操作 / 真实健身房场景？
较符合：
- 组表格是大按钮块状控件；勾选完成是明确的单点动作。
- 组间计时弹窗支持“最小化 + 震动”，适配把手机放一旁、靠触觉提醒。
- “训练进行中”悬浮锚点：允许切应用（比如查动作视频/回消息）而不丢状态。

## 5) 是否支持高频操作 / 是否存在断流感？
**高频操作强项**在“输入与确认”层：
- 自定义键盘 + 快捷增减 + 加一组：为高频数据录入优化。
- 勾选完成（✓）是明确的“commit”动作，形成节奏感（做完一组→打勾→休息）。

**断流风险**在“跨模块加载与重页面”层：
- 动作视频/体测等模块存在加载等待；
- 训练主链路之外的功能太多，容易把用户从训练 flow 拉走。

# Part 4 — 状态机推测（非常重要）

下面给出一个更贴近它交互的状态机（含最小化与组间有氧）：

```text
app_idle
  → (start_workout) workout_draft
  → (resume_workout) workout_active

workout_draft
  → editing_title
  → selecting_exercise
  → workout_active

workout_active
  → selecting_exercise
  → editing_set_cell
  → set_committed (✓)
  → rest_timer_running
  → inter_set_cardio_running
  → minimized (训练进行中 pill)
  → workout_completed (tap 完成)

editing_set_cell
  → numeric_pad_open
  → apply_quick_adjust (+/- / 加一组 / 热身组 / 递减组 / RPE)
  → back_to_workout_active

rest_timer_running / inter_set_cardio_running
  → minimized
  → completed_timer → workout_active

workout_completed
  → summary_review (推测：统计/分享/写心得/存模板)
  → sync_pending → synced
```

## 哪些状态是客户端即时状态？
- 训练计时（00:00→00:04）、组完成勾选、键盘输入、组间计时/震动、最小化后的“训练进行中”提示：都必须是**纯客户端实时状态**，否则体验会崩。

## 哪些可能依赖服务端？
- 动作库内容、动作图示/视频讲解、历史/图表、订阅状态、外部同步开关：更像是服务端/云端数据。
- 多处 Loading 说明其“数据可用性”并非总在本地命中。

## 是否可能使用 optimistic update / 本地缓存？
很可能：
- 训练中每一次输入都需要立即可见，典型做法是本地先写（optimistic），后台再同步。
- “训练进行中”跨页面/跨 app 的恢复，强烈暗示本地持久化（SQLite/Room/Realm 等）。

## 是否可能采用 append-only 数据流？
训练场景极适合 event log：
- 每次操作（新增动作/新增一组/修改重量/勾选完成/休息开始结束）都可以作为事件追加；
- 好处：可恢复、可回放、可做撤销/纠错、可做冲突合并（多端同步）。
从录屏可见的“强实时 + 可恢复”，符合 append-only 的收益模型。

# Part 5 — 工程架构逆向分析（为什么流畅/不流畅）

## 技术栈风格推测（非定论）
从 UI 形态看，它更像**跨端框架（Flutter/React Native）或深度自绘的原生**：
- 大量统一圆角卡片、统一按钮风格、统一弹层样式（跨端常见优势：一致的 design token）。
- 自定义数字键盘、环形计时器弹窗、径向快捷菜单：偏“自绘组件”思路，而不是纯系统控件堆叠。

## 是否是 streaming UI / skeleton strategy？
录屏体现更像“传统页面加载”：
- 出现明确的 Loading 遮罩/黑底进度，不像 streaming（渐进填充）；
- skeleton 不明显（更像等待后一次性渲染）。

## optimistic rendering / hydration blocking？
训练页内的编辑体验是即时的（偏 optimistic），但内容模块（体测/视频）有阻塞加载（偏 blocking）。
如果是 RN/Flutter：
- 训练页局部组件更新会很快；
- 但网络资源加载/首屏构建可能会产生明显的“整页遮罩”。

## “大组件问题”与潜在性能瓶颈
训练页是一个典型“超高交互密度”的复合页面：
- 顶部计时 + 标题输入 + 动作卡片 + 组表格 + 键盘 + 多弹层
若状态管理不谨慎（例如把整个页面绑定到一个大 store），容易造成：
- 输入一个数字导致整页重建（Flutter build 全树）或 RN 大量 re-render；
- 高刷手机下更容易感知卡顿。

## 为什么它在训练页“感觉可用”？
关键不在动画，而在**交互闭环设计**：
1. 数据输入被“自定义键盘”接管 → 让路径稳定、减少系统键盘抖动/格式校验复杂度
2. 勾选完成是单一 commit → 降低“我到底记没记上”的不确定性
3. 最小化与悬浮回流 → 降低中断成本（心理上更敢离开）

# Part 7 — 可借鉴模块（重点）

## 可以直接借鉴（几乎可原样吸收）
1. **“训练进行中”悬浮回流锚点**  
   - 本质：全局 session presence，让用户敢切后台/切页面而不怕丢训练。
2. **组表格的“勾选完成”commit 交互**  
   - 把每组做成一个明确完成事件，节奏感强、错误率低。
3. **组间计时弹窗（可震动 + 可最小化）**  
   - 训练中最常见的“等待”就是休息；把等待做成可控的小工具，而不是流程阻塞。
4. **一屏内完成“输入+微调”的自定义数字键盘**  
   - 特别是：加一组、一键改（+/-）、kg/lbs 切换，这些都属于高频刚需。

## 需要改造后借鉴（理念可用，但实现不适配 FitCoach）
1. **径向快捷创建菜单（新建训练/HIIT/休息/体测…）**  
   - FitCoach 可保留“快捷入口”理念，但菜单项应围绕“训练连续流”，避免把用户带去长尾模块。
2. **“工具箱式键盘”里的高级概念（热身/递减/RPE）**  
   - FitCoach 应把它们做成“预测/自动建议”的结果（默认自动生成 warmup/drop set），而不是把选择权全抛给用户。
3. **“今日安排/周期结束/模板”这类计划结构**  
   - 计划可作为“推荐背景”和“自动推进依据”，但不要让用户为了开练先做计划决策。

## 不应该学习（会破坏 FitCoach 核心理念）
1. **过度“全家桶化”的模块堆叠（饮食/经期/体测/多平台同步）作为主导航中心**  
   - FitCoach 的护城河是“训练即时记录系统”，全家桶会稀释产品心智与首页流式体验。
2. **重页面加载 + 遮罩 Loading 的信息架构**  
   - 这会直接制造“断流感”，与 FitCoach 的 streaming/island 方向相反。

# Part 8 — FitCoach 下一步升级建议（可落地）

> 目标：把“训记做对的训练中闭环”吸收进来，但用 FitCoach 的 instant-shell + streaming + predictive set 把它做得更轻、更连续、更高刷。

## UX 升级建议
1. **全局 Session Presence：训练进行中 Pill（可拖拽/可折叠）**
   - 首页/任何页面保持一个“继续训练”入口（显示计时/当前动作/下一组），点击即回到 runtime。
2. **组完成的“单手 commit”强化**
   - 每组行提供：单击 ✓ 完成、长按撤销、左右滑动快速复制上一组/增加一组（减少点键盘的频率）。
3. **休息体验做成“可忽略但可依赖”的工具**
   - 休息计时：弹窗/灵动条 + haptic；允许最小化；结束时震动/声音可配置。
4. **把“输入”变成“确认”**
   - 默认展示 prediction 的下一组；用户仅需要：
     - 点一下确认完成（✓）
     - 或轻微微调（+2.5kg / +1 rep）

## 架构升级建议（结合 instant-shell / streaming / island）
1. **Workout Runtime 独立于页面：以“运行时核心”驱动 UI（instant-shell）**
   - runtime store 常驻（内存 + 本地持久化），页面只是投影（island）。
2. **训练页组件拆岛：Timer / CurrentExercise / SetTable / RestOverlay 各自 island**
   - 避免一个 set cell 更新引发整页重渲；保证高刷。
3. **数据层用 event log（append-only）做本地优先**
   - UI 写入事件（SetCompleted/SetEdited/ExerciseAdded/RestStarted…）立即生效；
   - 后台聚合成“当前训练快照”和“历史记录”，并异步 sync。

## Runtime 升级建议（高刷 + frictionless logging）
1. **预测组系统与快捷微调并存**
   - 训记用“键盘工具箱”解决输入；FitCoach 用“预测+微调”减少输入。
2. **离线优先 + 冲突可合并**
   - 训练中任何时刻网络抖动不应影响可用性；同步失败只影响云端，不影响现场记录。
3. **输入控件的性能预算**
   - SetTable 虚拟化/局部刷新；
   - 键盘与表格之间避免 layout thrash（尤其移动端软键盘弹起）。

## 状态机升级建议
1. **显式状态机（而不是隐式 UI flag）**
   - idle → active → minimized → resumed → completed → synced
   - editing_set_cell / rest_running / pending_sync 等子状态清晰化
2. **“完成”不是唯一 commit**
   - 每个 set 的 ✓ 都是 commit；完成只是“session close”。这能显著降低“忘记点完成导致数据丢失”的焦虑。

## 首页流式体验升级建议
1. **首页 streaming：先渲染“继续训练/最近动作/预测下一步”，再补齐长尾内容**
2. **progressive load recommendation** 与“今日训练进度”融合
   - 训记用 20% 进度驱动；FitCoach 可用“下一步动作/下一组”来驱动，更贴近现场。

## 训练记录流升级建议
1. **Append-only 训练流（类似 Timeline）**
   - 用户看见的是：动作A 第1组✓ → 休息 → 动作A 第2组✓ …（连续记录流）
2. **历史复用做到“零选择”**
   - 默认带出上次训练的动作顺序与重量区间（甚至 warmup 自动插入），用户只需开始做并打勾。

# Part 9 — 最终结论（非常重要）

**一句话：为什么这个竞品成立？**  
它成立在“训练场景下最关键的三件事”做对了：**输入路径稳定（自定义键盘）+ 完成节奏清晰（✓）+ 状态不丢（训练进行中回流）**，让用户在真实健身房里保持心理连续性。

**一句话：为什么 FitCoach 可以做得更强？**  
FitCoach 若用 **instant-shell + streaming/island + append-only runtime + predictive set** 把“输入”进一步压缩为“确认/微调”，并把一切阻塞与跳转从训练流中剔除，就能在**高频交互、前端 runtime、状态可恢复与心理不断流**四个层面上，比“工具箱式重功能”路线更锋利、更快、更可持续。
