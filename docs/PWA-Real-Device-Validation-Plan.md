## 2. 安装流程验证（Install Flow Validation）

### 2.1 Android Flow（Chrome）
1. 扫码打开链接（首次访问）
2. 触发安装（系统 Banner 或「个人中心 → 添加到主屏幕」）
3. 从桌面图标打开（确认无浏览器 UI）
4. 进入任意页面（训练/饮食/历史/我的）确认底部导航正常
5. 彻底杀进程（任务管理器滑掉）
6. 从桌面再次打开（验证恢复能力）

**期望：**
- 图标正确、全屏、导航正常
- 不需要手动刷新
- 再次启动不会白屏/卡死

### 2.2 iOS Flow（Safari + A2HS）
1. Safari 扫码打开链接
2. 分享 → 添加到主屏幕
3. 从桌面图标启动（确认 standalone）
4. 锁屏/后台挂起 1~5 分钟 → 回到前台
5. 彻底杀进程 → 从桌面启动
6.（可选）重启手机 → 从桌面启动

**期望：**
- 视口不被刘海/底部条遮挡
- 缓存内容可读（至少 dashboard/history 可用）
- IndexedDB 不丢（见第 3 部分用例）

## 3. 离线测试（Offline Testing）

> 核心：验证“断网时仍可用、数据不丢、联网后能自动同步”。  
> 记录：每一步都截图/录屏（尤其是异常）。

### 3.1 训练离线保护（最关键）
场景：
1. 在线打开 App，确保已登录
2. 开始训练（进入 workout 流程）
3. 关闭网络（飞行模式或关闭 Wi‑Fi/蜂窝）
4. 记录多组训练（至少 3 个动作、每个 2~3 组）
5. 完成训练（触发本地保存与入队）
6. 离线状态下：杀进程 → 重新打开 App（仍离线）
7. 打开「历史」或训练相关页面，确认本地记录存在
8. 恢复网络
9. 观察顶部同步条：应自动同步并最终消失

**期望：**
- 无数据丢失
- UI 不冻结、不崩溃
- 联网后自动同步成功（无需手动刷新）

### 3.2 Dashboard / History 离线缓存
场景：
1. 在线打开 Dashboard + History（确保已加载）
2. 关闭网络
3. 从桌面重新打开（或刷新）

**期望：**
- 有内容（不是空白页）
- 仅显示离线提示，不阻塞使用

## 4. Service Worker 更新测试（Silent Update Validation）

> 目标：确认“发布后自动更新”，不会出现 **旧 JS/旧页面**、不会死循环重载。

场景：
1. 安装当前版本（Android Standalone / iOS A2HS）
2. 保持 App 打开在任意页面（尤其是 workout/训练中页面）
3. 发布一个**可见的小改动**（例如首页标题增加一个字符）到同一环境
4. 等待 1~2 分钟（当前实现每 60s `reg.update()`）
5. 观察：
   - 是否弹出更新提示（update banner）
   - 是否在非训练状态自动 reload 生效
   - 若在训练中：是否延迟到训练结束后 reload
6. reload 后确认 UI 已更新

**重点观察：**
- 更新后是否出现资源 404、白屏、无限 reload
- 是否出现“新 HTML + 旧 JS”导致 hydration 错乱

## 5. 移动端 UX 主观审计（App-like Feel）

每台设备至少评估（1~5 分）：  
- 触控响应（按钮命中率、无 300ms 延迟）  
- 滚动顺滑（历史列表/长页面）  
- 过渡顺滑（页面切换/动画）  
- 布局稳定（无明显闪烁/跳动）  
- 输入体验（键盘遮挡/光标定位）  
- 单手可达（底部导航好按）  

记录：**最影响“像 App”的 Top 3 痛点**。

## 7. Bug 收集 + 稳定化输出（Issue List）

> 建议统一用下面模板记录，每个 issue 必须可复现、可定位、可验证修复。

### 7.1 分级标准
- **Critical**：数据丢失/无法启动/无法训练/无限 reload/安装不可用
- **Major**：严重影响体验（键盘遮挡关键按钮、离线空白、同步长期失败）
- **Minor**：局部问题（个别页面样式错位、小概率 toast 异常）
- **Cosmetic**：纯视觉瑕疵（间距/对齐/文字）
